# EpicとWorkspaceの運用ルール (Workspace Management)

本ドキュメントは、You_Incシステムにおける「プロジェクト（Epic）」の管理方法と、関連するディレクトリの役割分担（ルーティング）を定義するアーキテクチャルールです。
すべてのAgentは、新規Epicの開始・進行・完了においてこのルールを遵守してください。

## 1. 役割の分離（インデックスと実体）

プロジェクトの進行においては、情報が散逸しないよう「概要とポインタを持つインデックス」と「設計と議論を持つ実体（SSOT）」を明確に分離します。

### 📄 `agent-core/epics/<epic_name>.md` (The Index / ルーティングハブ)
*   **役割**: プロジェクトの目的と概要、現在の大まかなステータスを示し、詳細情報へのポインタ（ルーティング）を提供するインデックスです。
*   **命名規則**: `systematic_trading.md` のように、内容を表すシンプルなスネークケースを使用してください。**連番プレフィックス（例: `01_`）は禁止**です。Epicは動的に生成・削除されるため、連番による管理は破綻します。
*   **持つべき情報**:
    *   Epicの目的と全体像（コンテキスト）。
    *   Phaseレベルのざっくりとした進捗ステータス。
    *   詳細な決定事項やTODOについては、必ず `workspaces/` を見に行くように促すリンク・指示。
*   **禁止事項**: ここに詳細な設計議論や、日々の細かなTODOリストを書いてはいけません。

### 📂 `agent-core/workspaces/<Epic名>/` (The Core SSOT / 実体)
*   **役割**: プロジェクトに関する「決定事項」「設計根拠」「進捗」をすべて集約管理するSSOT（唯一の真実のソース）となる「作業場」です。
*   **構造**: Agentが迷わず情報を取得できるよう、フラットではなく以下のディレクトリ構造とエントリーポイントを必ず持ちます。

```text
workspaces/<Epic名>/
├── _index.md        # 📍 エントリーポイント（ルーティング指示と概要）
├── docs/            # 📖 The SSOT (設計・決定事項)
├── tasks/           # 📋 The Tracker & RAM (進行管理と動的コンテキスト)
│   ├── progress.md  # 💽 The HDD (静的なタスク進捗の正本)
│   └── context.md   # 🧠 The RAM (動的な議論の現在地・ワーキングメモリ)
└── scratch/         # 🗑️ The Sandbox (一時領域・ゴミ箱)
```

*   **持つべき情報**:
    *   **`_index.md`**: Agentが最初に読むべきファイル。「設計書は `docs/00_...` から読め」「進捗は `tasks/progress.md` を見ろ」というAgent向けの読み順とルーティング指示を記述する。
    *   **`docs/`**: `00_Design_Decisions.md` 等の設計の決定事項、議論のプロセス、アーキテクチャの根拠。
    *   **`tasks/`**: 
        *   `progress.md`: 具体的な進捗状況やアトミックなTODOの管理（タスク完了時の低頻度更新）。
        *   `context.md`: 現在の焦点や最新の合意事項を50行以内で要約するワーキングメモリ（議論ごとの高頻度更新）。新規作成時は `agent-core/templates/Workspace_Context_Template.md` を使用すること。
    *   **`scratch/`**: 一時的なスクリプトや、Agent同士のレビューレポート、エラーログなどを置く。**この中のファイルはフェーズ完了時に破棄すること。**
*   **終了時の扱い**: プロジェクト（Epic）が完了した際、このワークスペース内に残された `docs/` などの有益な「普遍的ルール」や「アーキテクチャ図」は `agent-core/docs/` 等に昇格・移動させます。その後、このワークスペース自体はクリーンアップ（削除）可能となります。

## 2. Epicの終了とクリーンアップ (Standard Close Operation)
プロジェクトが完了した際の標準的なクローズ作業は以下の手順で実行し、「Leave No Trace（ステートを持たない）」の原則を徹底します。

1. **リソースの退避**: ワークスペース内の有益な資料や成果物を `second-brain/` や `agent-core/docs/` 等の恒久的な保存領域へ移動します。
2. **完了状態への更新**: `epics/<Epic名>.md` のステータスを `completed` に変更し、全タスクをチェックします。
3. **Git完了断面の保存**: この段階で一度 Git コミットを行い、「プロジェクトが正常に完了した」という断面（歴史）を記録します。
4. **ディレクトリの削除**: 退避完了後、`workspaces/<Epic名>/` ディレクトリを完全に削除します。
5. **Epicファイルの削除**: 履歴はGitに残っているため、インデックスファイル自体（`epics/<Epic名>.md`）も削除します。
6. **Gitクリーンアップ断面の保存**: 以上の削除状態を再度 Git コミットとして記録し、システムをクリーンな状態に戻します。
