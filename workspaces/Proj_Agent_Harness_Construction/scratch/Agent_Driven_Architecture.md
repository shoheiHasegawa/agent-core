# Zettelkasten駆動型 Agent-Flow 運用SOP

この文書は、Antigravity 2.0系における「人間（PO）とマルチエージェント」による非同期開発ワークフロー（Agent-Flow）の標準作業手順（SOP）を定義する。

## 1. パラダイムシフト
開発体制は、「人間がプロンプトで指示し、Agentが補佐する（Copilot型）」から、「**人間が仕様（What）を定義し、Agentが自律的に実装（How）を行う（Agentic型）**」へ完全に移行する。

人間の最大の責務は「ビジネス価値の創出」と「品質ゲートへの要件投下」であり、Agentの責務は「仕様を満たすコードの生成」「テスト」「知識の蒸留」である。

## 2. 状態遷移（ディレクトリ・ステートマシン）のフロー
`life-automation` 等の実行リポジトリにおいて、タスクは `.gemini/tasks/` 配下のディレクトリ移動によって状態遷移する。Agent同士の通信はファイルシステム（非同期イベントバス）を介して行われる。

### フェーズ 1: 要件定義と品質ゲート (Draft)
1. **人間**: 実現したい要件をMarkdownの仕様書として書き、`.gemini/tasks/draft/` に配置する。
2. **The Architect (AI)**: `draft/` の更新を検知して起動。`second-brain` の憲法やポリシー（Why）と照らし合わせ、エッジケースの漏れや矛盾をレビューし、ファイルにコメントを追記する。

### フェーズ 2: 実装とテストの自律ループ (Ready)
1. **人間**: Architectのレビューを通過した「完璧な仕様書」を `.gemini/tasks/ready/` へ移動（mv）させる（これが実装へのGOサインとなる）。
2. **Inspector & Developer (AI)**: `ready/` への配置を検知して起動。Inspectorがテストハーネスを作り、Developerが実装を回す。
3. *フェールセーフ*: エラー解決の自己リトライは最大3回まで。解決しない場合は人間にエスカレーションする。

### フェーズ 3: 自律バージョン管理と知識のハンドオーバー (Review & Done)
1. **The Reviewer (AI)**: テストをパスしたコードを監査し、DoDを満たしていれば自律的にバージョン管理システムへコミット（Fast-track）する。
2. **The Reviewer (AI)**: 実装ログから本質を抽出した「納品書 兼 収穫レポート（Harvest Report）」を作成し、工場の領土である `.gemini/outbox/` へ出力する。※工場が頭脳側のパスを知ることは憲法違反（Leaky Abstraction）であるため、自陣に出力して終了する。
3. **The Reviewer (AI)**: 仕様書を `.gemini/tasks/done/` へ移動させ、工場の処理を完了する。

### フェーズ 4: 知識の検疫と蒸留 (The Editor & The PO)
1. **The Editor (AI)**: 各工場の `.gemini/outbox/` を定期監視（Pull）し、新規のHarvest Reportがあれば自身の `second-brain/00_Inbox/01_Review_Queue/` 等の検疫エリアへ移動させる。
2. **The Editor (AI)**: レポートをZettelkastenフォーマットに整え、既存の知識とリンクさせる（※この時点ではまだPermanentには入れない）。
3. **The PO (人間)**: 検疫エリアにある知識をレビューし、承認（Merge）した瞬間にのみ `30_Resources/Permanent/` へと昇格させる。AIによるPermanentへの直接書き込みは厳禁とする。

## 3. 責務の境界（Why と How の分離）
- **`second-brain` (Why/Strategic)**: 「なぜ自動化するのか」「AI運用方針」「セキュリティ原則」を管理する。実装の詳細（言語やフレームワーク）には依存しない。
- **`life-automation` (How/Tactical)**: 「実装ルール」「ディレクトリ構成」「テスト手順」を管理する。

Agentは、この境界線をメタ視点で読み取りながら作業を行うことで、レポジトリ間の密結合（Leaky Abstraction）を防ぐ。

## 4. ルーターパターン（自己認識とエスカレーション）
Agent-Flowにおけるコストと精度の最適化戦略として、**「AIの動的ルーティング」**を採用する。
- **フロントライン**: 高速・低コストなモデル（Frontline Model）がDeveloperとして日常のコーディングとテストを担当する。
- **エスカレーション (サーキットブレーカー)**: Developerのエスカレーションは、AIのメタ認知（主観的な自己認識）に依存せず、**客観的なシステム・サーキットブレーカー**（例：同一タスクでのテスト失敗が3回連続した、Lintエラーが解消できない等）によって強制的に発火する。システムが限界を検知した時点で、タスクは `draft/` に差し戻され、人間と高推論なThe Architectへルーティングされる。
- **バックエンド**: 人間は差し戻しを検知し、高推論モデル（Backend/Architect Model）をThe Architectとして召喚し、設計の壁打ちを行って再びフロントラインモデルに実装を委ねる。

## 5. テストの階層化（AIの局所的最適解への防御）
AI特有の「目の前のテストを通すためだけに、アーキテクチャを静かに破壊する（Specification Gaming）」リスクを防ぐため、テストは以下の3層で構築し、DoDとして強制する。
1. **Contract / E2E Level**: ユースケース層の「In-Outの契約」を保証する仕様化テスト。AIはこの契約を無断で変更できない。
2. **Architecture Level**: DIコンテナの配線や依存関係の向きを検証するテスト。AIのバイパス（スパゲッティ化）を物理的に防ぐ。
3. **Unit Level**: エッジケースやバリデーションを検証する単体テスト。AIに境界値テストを網羅的に量産させる領域。

**【権限の非対称性 (RBAC) によるチート防止】**
AIが「テストを通すのが難しいため、上位のテストファイル自体を書き換える（チート）」という最悪のハックを防ぐため、テストの階層化は権限の階層化と同期させる。
- **上位レイヤー (Contract / Architecture)**: 人間またはThe Architectが記述し、Developerにとっては**【Read-Only（不可侵）】**とする。
- **下位レイヤー (Unit / Implementation)**: Developerが自由に作成・改変してよい。
これにより、Developerは上位のテスト（契約）を書き換えることが物理的に不可能になり、正しく実装コード側を修正するよう強制される。
