# 13_Event Driven Batch Architecture

このドキュメントでは、You, Inc. アーキテクチャにおける非同期バッチ処理（特に `00_Inbox` の回収等のイベント駆動型バッチ）の設計方針と、コンポーネント間の責務分離（DI: 依存性注入）について定義します。

## 1. コンセプト: イベント駆動とDIによる疎結合

ファイルの移動や外部APIとの通信など「副作用」を伴うI/O処理において、Agentが直接パスをハードコードしたり同期的に通信し合ったりすると、システムは密結合になり、同期遅延やコンフリクト（競合）に弱くなります。

これを防ぐため、バッチ処理は以下の原則に従います。
- **DI (Dependency Injection)**: 実行に必要なパスやクレデンシャルは、上位層（`agent-core`）から下位層（`core-service`）へ実行時に注入する。
- **イベント駆動 / 非同期Queue**: Agent間やディレクトリ間のデータ受け渡しは、同期APIではなく「特定のディレクトリ（Queue）へのファイル配置と移動」をトリガーとする。

## 2. 責務の分離

バッチ処理は、オーケストレーターと実行エンジンの2つのコンポーネントに分離されます。

### 2.1 `agent-core` (オーケストレーター / 設定・スケジュール)
- **スケジューリング**: Cronや常駐プロセス（ファイルウォッチ）等の「いつ実行するか」というトリガーを管理する。
- **設定とDI**: どこから（Source）どこへ（Destination）移動するか、というパス情報を設定ファイル（YAML等）に持ち、実行時に `core-service` のスクリプトへ引数として渡す。

### 2.2 `core-service` (実行エンジン / 汎用ツール)
- **ステートレスなI/O**: 与えられた引数（SourceとDestination）に従って、純粋に「ファイルを移動する」「APIを叩く」といったI/O処理のみを実行する。
- 自らの実行タイミングや、対象のパス（環境依存情報）を内部に保持しない。

## 3. 具体例: Inbox回収バッチと工程ごとのステート管理

`mobile-vault`（iCloud）等から投下されたファイルを安全に処理し、コンフリクトを防ぐため、`agent-core` 内に浅くシンプルな「状態遷移（工程）」を表すディレクトリ構成を設けます。

**ディレクトリ構成（ステートマシン）の例:**
- `agent-core/queue/pending/`: iCloudから回収した「未着手」のファイル置き場。
- `agent-core/queue/processing/`: 現在Agentが掴んで「処理中」のファイル置き場。
- `agent-core/queue/error/`: 処理に失敗した際の「Dead Letter Queue」。
- `agent-core/queue/done/`: 処理完了のアーカイブ。

**処理フロー:**
1. **投下**: ユーザーがiCloudの `00_Inbox` にタスク（Markdown）を作成する。
2. **回収（Pending）**: `agent-core` のスケジューラが、iCloudの `00_Inbox` から `agent-core/queue/pending/` へファイルを移動（`mv`）させる。この時点でiCloudの干渉（同期遅延・コンフリクト）から完全に切り離される。
   - **冪等性（重複防止）の担保**: 異なるファイルシステム間（iCloud等）の `mv` は内部的に非アトミック（cp+rm）であるため、移動直後のクラッシュ等で二重処理されるリスクがあります。移動時にファイル名に `Task_ID` (タイムスタンプ等) を付与することで、同名タスクの重複処理（二重PR作成など）をシステム的に防ぎます。
3. **処理開始（Processing）**: タスクを担当するAgentが起動し、対象ファイルを `queue/processing/` へ移動させてからパース・推論を始める。このアトミック移動自体が排他制御（ロック）となり、他Agentの二重取りを防ぐ。
4. **完了・エラー**: 処理に成功すれば `done/`、YAMLエラー等で失敗すれば `error/` へ移動させる。

## 4. セルフヒーリング（自己修復）によるゾンビ化対策

処理中にAgentのプロセスがダウンしたり、PCが再起動した場合、ファイルが `queue/processing/` 内に取り残される「ゾンビ化」リスクがあります。
これを防ぐため、複雑な死活監視プロセスを設けるのではなく、「起動時のセルフヒーリング」によって冪等性を担保します。

- **起動時リトライ**: `agent-core` のバッチまたはAgent起動時、`queue/pending/` を確認する前に、必ず自身の `queue/processing/` を走査します。
- **回収と再開**: そこに残存ファイルがあれば、「前回処理中に落ちた未完了タスク」として扱い、再度 `queue/pending/` に戻すか、そのまま処理をリトライ（再開）します。
- **ポイズンピル（無限クラッシュ）対策とDLQ隔離**: もしそのファイル自体が原因でAgentがクラッシュしていた場合、リトライが永遠に繰り返されるデッドロック（Poison Pill）になります。これを防ぐため、リトライ時にファイル名へ回数を付与（例: `retry1_xxx.md`）し、上限（例: 2回）に達したものは即座に `queue/error/` (Dead Letter Queue) へ隔離します。
