# 99_CI/CD Pipeline Architecture

このドキュメントでは、You, Inc.のコアリポジトリにおけるCI/CDパイプラインと品質ゲートの要件、および障害発生時の「フェールセーフと人へのフォールバック」戦略について定義します。

## 1. コンセプト: 適応型ガバナンスとフェールファスト

Agent駆動のアーキテクチャでは、Agentの「暴走（ハルシネーション）」や「無限リトライ」を防ぐことが最重要です。
そのため、以下の原則を各リポジトリに適用します。

- **ローカルLintとAuto-fixの徹底**: コミット・Push前に `pre-commit` フックにより構文チェックを実施する。運用摩擦を下げるため、`prettier` 等による **「自動修正（auto-fix）」** を強制し、軽微なエラーはAgentが自律的に修正してからコミットできる体制とする。
- **YAMLスキーマのランタイムバリデーション**: YAMLフロントマターが暗黙のコントラクトになる脆弱性を防ぐため、Agentがファイルを読み書きする際は生のI/Oを禁じ、`core-service` に実装されたバリデーションモジュール（Zod/Pydantic等）を必ず経由させる。
- **Direct Pushの完全禁止とAuto-Merge化**: CI事後失敗時の `main` 汚染（カスケード障害）を防ぐため、Agentによる `main` への Direct Push は完全に禁止します。「PR必須 ＋ CI通過時の GitHub Auto-Merge 機能」を利用し、人間のクリックなしでの自動反映（UX）とクリーンな状態を両立させます。
- **Human-in-the-loop (HITL)**: 重大なエラーやコアルールの変更時は、必ず人間（社長）へエスカレーションして承認や手動介入を仰ぐ。

## 2. タスク粒度の分離とステートレス化 (リソース最適化)

Agentが人間のPR承認（Approve）をスリープして待ち続けると、システムリソースが枯渇（デッドロック）します。これを防ぐため、以下のステートレスなライフサイクルを定義します。
- **タスクの完了＝PRの作成**: AgentはPRを作成した時点で「タスク完了」とみなし、プロセスを安全に終了（Exit）してリソースを解放する。
- **イベント駆動での再始動とステート復元**: 社長がPRをマージしたことや、CIエラーで修正を求められたことを Webhook 等で検知し、別Agentプロセスを起動する。この際、ステートレスなAgentが「今何回目のリトライか」を喪失し無限ループに陥るのを防ぐため、GitHub PRのラベル（例: `retry: 1`）やコメントにメタデータを保存し、起動時に現在ステートを復元します。

## 2. 各リポジトリのCI/CDパイプライン要件

リポジトリの責務に応じて、CIの厳格さと手動承認の有無（Quality Gate）を切り替えます。

### 2.1 `second-brain` (知識層 / SSOT)
- **Pre-commit (Local)**: Markdownのフォーマットチェック・整形（prettier, markdownlint等）。
- **Actions (Remote)**:
  - **YAMLフロントマター検証**: 必須スキーマ（`type`, `status`, `created`, `requires_human` 等）の厳密チェック。
  - **内部リンク・整合性チェック**: デッドリンクチェックを行い、ナレッジグラフの破損を防ぐ。
- **承認ゲート**: 運用負荷低減のため **Direct Pushを許可**。ただし、上記のCIに違反した場合は即座にリジェクトされる。

### 2.2 `agent-core` (オーケストレーション層)
- **Pre-commit (Local)**: Python/TSの静的型チェック、Linter、軽量なユニットテスト。
- **Actions (Remote)**:
  - **ワークフローとテンプレート検証**: `workflows/` のルーティング整合性や、`templates/` のパーステスト。
  - **隔離実行環境のセーフティテスト**: `git worktree` やDockerによる隔離が機能し、ホストOSを汚染しないかのチェック。
- **承認ゲート**: `AGENT.md` などコアルールの変更は Agent単独でのマージを禁止し、**必ず社長（人間）のApproveを必須** とする。

### 2.3 `core-service` (DDD/ツール層)
- **Pre-commit (Local)**: Domain/Infra層の網羅的なユニットテスト高速実行、Docstring・型の厳密な静的解析。
- **Actions (Remote)**:
  - **仕様化テスト (E2E相当)**: `tests/integration/` にてApplication層の振る舞いを検証し、仕様書とのトレーサビリティを確保。
  - **独立Agentによる静的解析**: レビュアーAgentによるDDDレイヤー違反（依存方向の逆転等）のチェック。
- **承認ゲート**: Direct Push完全禁止。CIとレビュアーAgentのパス後、**人間（社長）の最終承認（HITL）が必須**。

### 2.4 `mobile-vault` (iCloud / フロントエンド)
- **Pre-commit (Local)**: モバイルからの操作が主のため実質不要。
- **Actions (Remote)**:
  - `10_Private_Notes/` へのAgent不可侵監査。
  - ダッシュボード（`20_Dashboards/` 等）への出力ファイルフォーマット検証。

## 3. エラー時の「人間へのフォールバック」フロー

CIパイプラインでエラーが発生した場合、またはマージコンフリクトが発生した場合、システムは以下のフローで安全側に倒れます（フェールファスト）。

1. **即時ブロック**: CIエラーやコンフリクトを検知した時点で、処理（Push/Merge）を即座にブロックする。
2. **Post Mortem（事後検証）の作成**: エラーを検知したAgentは、`Incident_Recovery_Template.md` を用いて「エラーの経緯、原因、ログ」をドキュメント化する（自動修正の無限ループを防ぐため、Day 1では自律復旧を深く追求しない）。
3. **エスカレーション**: インシデントレポートをPRのコメント、または `Error_Inbox/` に起票する。
4. **凍結と待機**: 対象のEpic/Workflowを一時凍結（Grace Period）し、人間の介入を待つ。
5. **Human-in-the-loop**: 人間（社長）がログを確認し、手動で修正するか、`Inbox` から新たな指示を与えて再始動させる。
