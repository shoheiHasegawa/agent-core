# 30_core-service アーキテクチャ設計書

## 1. 概要と責務
`core-service` は、会社全体を支える「情報システム部・道具箱」である。
外部API通信やDB操作といった「副作用」を本リポジトリ内にカプセル化しつつも、Agentの「状態（State）」は一切持たない**ステートレスで堅牢なサービス**の集合体である。誰が使うか（Agentか人間か）を知らない純粋なドメインロジックと、それに付随する安全なI/Oを提供する。（DIP: 依存性逆転の原則の徹底）

## 2. ディレクトリ構成
DDD（ドメイン駆動設計）のレイヤードアーキテクチャを採用し、ユースケース（機能群）ごとに階層化する。

```text
core-service/
├── README.md               # [薄いルーター] 全体概要と各ルールへのポータル
├── INDEX.md                # [オンデマンド] 詳細なディレクトリマップ（地図）
├── docs/                   # ペルソナ別・自己記述型ドキュメント
│   ├── architecture.md
│   ├── rules/              # [JITロード] タスクごとに分割された詳細ルール群
│   │   ├── ddd_guidelines.md
│   │   └── api_gateway.md
│   └── review.md
├── src/
│   ├── domain/             # ドメインモデル・純粋なビジネスルール (Docstringで仕様担保)
│   ├── application/        # ユースケース・ドメインのオーケストレーション
│   │   └── task_management/
│   │       ├── task_service.ts
│   │       └── task_service_spec.md  # ★ユースケース単位の仕様書 (SDD)
│   └── infrastructure/     # 外部IFの具象実装（副作用は持たないAdapter）
└── tests/
    ├── unit/               # Domain/Infra 等の複雑なエッジケース・網羅的テスト
    └── integration/        # Application層の振る舞いを固定する仕様化テスト (e2e相当)
```

## 3. コア機能とルール

### 3.1 クリティカル処理のService化と副作用の封じ込め
「外部API通信」や「DB操作」など、品質の安定性が強く求められる処理は、Agentに都度スクリプトを書かせるのではなく、本リポジトリ内にServiceとして堅牢に実装する。これによりAgentの作業品質のばらつきを回避する。

### 3.2 SDD (Specification Driven Development) と Application層の仕様化
システム全体の振る舞いのエントリーポイントである **Application層（Service）** にのみ `spec.md`（仕様書）を配置する。
Domain層やInfrastructure層の詳細はコード内のDocstringで仕様を担保し、ドキュメントの二重管理を防ぐ。

### 3.3 依存性注入 (DI) によるシークレットの分離
`core-service` はAPIと通信する機能（Gateway等）を持つが、APIキーなどの**シークレット情報は一切保持しない**。
また、Webサーバーとして常駐するわけではないため**コンテナ化（Docker）は不要**であり、純粋なPythonライブラリ群として実装する。
実際のクレデンシャルは、呼び出し元である `agent-core` 側で `sops` 等を用いて暗号化管理され、実行時（`pyproject.toml` を介したローカルパスでのDI等）に注入される構造とする。

### 3.4 仕様とテストのトレーサビリティ (Traceability)
仕様書（`spec.md`）とテストコード（`tests/`）の乖離を防ぐため、仕様書内に一意のシナリオID（例: `[SCENARIO-01]`）を定義する。
テストコードの関数名やDocstringに必ずそのタグを直接記載させることで、「このテストはどの要求仕様を担保しているか」というマッピング情報をコード側に持たせる。

### 3.5 自己記述型リポジトリの徹底とAgentへのルール強制（JITロード）
Agentのコンテキスト圧迫と情報の忘却（Lost in the middle）を防ぐため、「ルールの遅延ロード（Lazy Loading）」を徹底する。
1. **ルーティング（常時ロード）**: Agentがタスクを開始する際、Workflowの最初のステップとして「作業前に必ず対象リポジトリの薄い `README.md` を読み込むこと」を強制する。
2. **JITロード（遅延ロード）**: `README.md` には、そのリポジトリ専用の「ルールの在処（ルーティングテーブル）」が定義されている（例: 本リポジトリなら `docs/rules/`）。Agentはこれに従い、自身のタスクに関連する詳細ルール（例: インフラ層の開発なら `docs/rules/api_gateway.md`）**だけ**をピンポイントで読み込み、実装に移る。巨大な一枚岩のルールファイルを読ませることは禁止する。

### 3.6 厳格なマージ・デプロイメントフロー (品質ゲート)
`core-service` はシステムの中で最も副作用（DB操作・外部API連携）が集中するクリティカルなリポジトリであるため、他のリポジトリとは異なる最高レベルのガバナンス（品質ゲート）を適用する。

1. **Direct Push の完全禁止**: `core-service` における Agentの `main` ブランチへの直接Pushはいかなる場合も禁止。必ず Pull Request を作成する。
2. **静的解析と儀式の強制**: `ruff` 等の静的解析ツールと `pre-commit` フックを導入し、DDDレイヤー間の不正参照や構文エラーを**コミット前に機械的にブロック**する。
3. **自動CIゲート (仕様化テスト)**: PRが作成されると、GitHub Actions等のCIで `tests/integration/` の仕様化テストが必須で走る。
4. **レビュアーAgentによるセマンティックレビュー**: 静的解析は機械(ruff)に任せ、独立した「レビュアーAgent」は「仕様書(SDD)の要件を満たしているか」等の意味論的レビューのみに専念する。
5. **社長（人間）の最終承認 (Human-in-the-loop)**: 上記の全ゲートをクリアしたPRのみが社長へエスカレーションされる。社長の `Approve` がない限り、本番環境へのマージ・デプロイは絶対に実行されない。
