# 50_Obsidian View Architecture

このドキュメントでは、You, Inc. のアーキテクチャにおけるフロントエンド（View層）としての Obsidian の責務と設計方針を定義します。

## 1. コンセプト: フロントエンドとしてのObsidian

従来のPARAメソッド等において、Obsidianは「データ（Markdown）」と「設定（プラグイン等）」が混然一体となったモノリスとして扱われがちでした。
You, Inc. アーキテクチャでは、**Obsidian を純粋な「UI / View層」として再定義** し、バックエンド（`second-brain` のデータ）や コントローラー（`agent-core`）と明確に分離します。

## 2. 責務の分離と `90_Meta` への内包

Obsidianがフロントエンドとして機能するための「設定ファイル」や「ダッシュボード用の動的Markdown」は、純粋な知識（ナレッジ）とは性質が異なります。
そのため、View層の構成要素はすべて **`second-brain/90_Meta/` 配下に内包・集約** して管理します。

### 2.1 管理対象の例
- **`.obsidian/` ディレクトリ**: プラグイン（Dataview, Templater等）、テーマ、ショートカットなどの設定ファイル全般。
- **ダッシュボード（Dashboards）**: 進行中のプロジェクトやタスクを俯瞰するための `Dashboard.md` 等。
- **テンプレート（Templates）**: ユーザー（人間）が新規ノートを作成する際に使用するスニペット類。

### 2.2 Agentの不可侵・責務境界
- **Agentの振る舞い**: Agentはナレッジの内容（Data）や YAMLフロントマター の書き換え（状態更新）を行いますが、**`.obsidian/` の設定ファイルやダッシュボードの表示ロジックには一切干渉しません**。
- **ViewとControllerの分離**: 「進行中のタスク一覧を出す」といった計算は、Agentが毎回Markdownとして出力するのではなく、Obsidianの `Dataview` 等を用いてView側で動的にレンダリングします。Agentの責務は、対象ファイルのYAMLに `status: in-progress` というフラグを立てる（状態を管理する）ことのみです。

## 3. デバイスごとのユースケース

Obsidianは、利用するデバイスによって役割が変化します。

- **PC向け (Rich View)**:
  - `second-brain` ディレクトリ全体を開くメインのワークスペース。
  - Dataview等を用いた高度なダッシュボードの閲覧や、ナレッジグラフの探索、全体俯瞰に用いる。
- **モバイル向け (Thin View)**:
  - `mobile-vault`（iCloud同期）のみを開く軽量なワークスペース。
  - 単純な `00_Inbox` へのメモ投下と、簡単な静的ビューの閲覧に特化させ、重いプラグインによる同期遅延やバッテリー消費を防ぐ。
