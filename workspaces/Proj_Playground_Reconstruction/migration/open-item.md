# Open Items (移行プロジェクト未決課題一覧)

## ✅ 解決済みの課題 (Resolved)

### 1. Docker隔離実行環境の配置と責務 (`agent-core` vs `core-service`)
- **決定事項**: 
  - **`core-service`**: コンテナ化は**不要**。常駐サーバーではなく純粋なライブラリ群として実装し、品質はテストコード（仕様化テスト）と静的解析（ruff）で担保する。
  - **`agent-core`**: カスタムDockerfileは作らず、エージェントが未知のコードを安全に実行する際のみ、公式の汎用イメージ（`python:3.x-slim`等）を `docker run --rm` で使い捨てサンドボックスとして呼び出すのみとする。

## ✅ 解決済みの課題 (Resolved)

### 2. レビューAgentの静的解析実装
- **決定事項**: `ruff`（フォーマッタ・Linter）を導入し、pre-commitフックでコミット前に強制（儀式化）する。LLMは静的解析を行わず、セマンティックなレビューのみを行う。

### 3. リンク切れ検知ツールの選定 (`second-brain`)
- **決定事項**: CI（Push後）では手遅れになるため、ローカルの pre-commit フックやエディタ側の拡張機能を用いて、コミット前に検知するアプローチへ変更。

### 4. クレデンシャルDIの経路
- **決定事項**: 既存の `life-automation` 同様、`sops` による暗号化を踏襲する。また、`agent-core` の `pyproject.toml` 等でローカルパスとして `core-service` を参照・DIする構成とする。

### 5. トランザクションデータから「知識抽出」するトリガーの定義
- **決定事項**: Global Constitution に定義されている「Harvest Reportの自律生成」ルールに従う。セッション完了時や議論の区切り等で、自律的に Harvest Report を生成し、Inbox（検疫エリア）へ出力する。

### 6. 並行稼働時における書き込み競合リスク
- **決定事項**: 現行で稼働している機能がないため、並行稼働テスト（Dry-run等）は行わない。移行完了までLegacyを居座らせ、完了次第切り替える方針とする。

### 7. `daily-tools` 内のUI・対話型依存ツールの扱い
- **決定事項**: そのまま `core-service` へ持っていくことはしない。何を捨てるか、どう分割するかは、Phase 02（分析タスク）の実作業の中で決定する。

### 8. Gitリポジトリ履歴の統合戦略（モノレポ化のコンフリクト）
- **決定事項**: 旧 `second-brain` 等の過去の Git 履歴はすべて破棄（全捨て）し、`You_Inc` として全く新しいコミット履歴から再スタートする。
