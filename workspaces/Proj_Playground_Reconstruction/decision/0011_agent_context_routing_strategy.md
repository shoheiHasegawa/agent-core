# ADR 0011: Agent Context Routing and Lazy-Loading Strategy (Agentのコンテキストルーティングと遅延ロード戦略)

## 1. Context (背景と課題)
You, Inc. のアーキテクチャ設計において、「Agentが各ディレクトリの責務やルールを守るための自己記述型リポジトリ」を採用した。しかし、Agentに対してすべてのルール（README、INDEX、詳細な実装ルール等）をタスク開始時に一括で読み込ませると、以下の深刻な問題が発生する。
1. **コンテキストウィンドウの圧迫**: トークン数の消費によるコスト増と処理速度の低下。
2. **Lost in the middle（情報の忘却）**: プロンプトが長大になることで、LLMの注意力が散漫になり、結果的に重要なルールを無視して暴走するリスク。

## 2. Decision (決定事項)
Agentのコンテキスト（前提知識やルール）の渡し方について、**「ルーティング層（常時ロード）」** と **「詳細ルール層（JIT/遅延ロード）」** を明確に分離する戦略を採用する。

### 2.1 起点の明確化 (`GEMINI.md` と `README.md`)
Agentのメタ知識（どこに何があるか）の起点として、ルート直下に以下の2つを配置し、Agent起動時に必ず読み込ませる。
- **`README.md`**: 【人間向け】リポジトリの全体像と目的。
- **`GEMINI.md`**: 【AI向け】グローバル憲法と、各ルールへの**ポインタ（ルーティング）**。
※これら起点ファイルには**「絶対に具体的なルールを書かない（FAT化防止）」**というメタ・ルールを徹底する。

### 2.2 INDEX.md のオンデマンド化
`INDEX.md` はディレクトリの地図（リスト）として機能させるが、最初からは読ませず、Agentが「対象のファイルが見つからない・全体構造を知りたい」と判断した時にのみ自律的に読みに行く（オンデマンドロード）運用とする。

### 2.3 詳細ルールの抽象度分離と JIT (Just-In-Time) ロード
巨大な一枚岩のルールファイルを作成することは禁ずる。ルールは抽象度に応じて以下の2箇所に分離して格納する。
1. **知的・ビジネス的制約 (Why/What)**: `second-brain/10_Areas/` に配置。
2. **機械的・システム的制約 (How)**: `second-brain/90_Meta/System_Rules/` に配置（RAGノイズからの隔離）。
Agentは、`GEMINI.md` のポインタに従い、自身のタスクに関連する詳細ルール**だけ**をピンポイントで遅延ロード（Lazy Loading）する。

### 2.4 ルール依存の単方向性（デッドロック防止）
ルールが複数箇所に分離されたことによる「循環参照（無限ループ）」を防ぐため、以下の依存グラフをシステム全体の規律とする。
- **許可**: `10_Areas` (ビジネス理念) が、実装の詳細を規定するために `90_Meta/System_Rules` (システム制約) を参照・リンクすること。
- **禁止**: `90_Meta/System_Rules` から `10_Areas` を逆参照すること。
システム制約（How）はビジネス理念（Why）に依存してはならず、この単方向性を維持することで、Agentがルールの優先順位に迷いフリーズ（デッドロック）することを原理的に防ぐ。

## 3. Consequence (結果)
- **メリット**: LLMのコンテキストが常にクリーンで軽量に保たれる。Agentが「今実行しようとしているタスクのルール」だけに強烈にフォーカスでき、ハルシネーションやルール違反が激減する。
- **デメリット**: ルールファイルが細分化されるため、人間側でのドキュメントの整理・メンテナンスコストが若干上昇する。
