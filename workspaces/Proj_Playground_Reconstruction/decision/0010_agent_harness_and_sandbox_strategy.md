# ADR 0010: Agent Harness and Sandbox Strategy (Agentの暴走防止と品質安定化の戦略)

## 1. Context (背景と課題)
You, Inc. のアーキテクチャ設計において、Agent駆動の運用を前提とした際、レビューア（サブエージェント）から以下の深刻な脆弱性が指摘された。
1. **物理的破壊のリスク**: Gitの隔離だけでは、Agentがローカル環境で `rm -rf` などを実行して他のリポジトリを破壊する事故を防げない。
2. **コアルールの改ざん**: Agentが自身を律する `AGENT.md` などのルールを勝手に書き換えてしまう恐れがある。
3. **作業品質のばらつきとI/Oの暴走**: 外部API通信やDBアクセスをAgentに都度スクリプトを書かせて実行させると、ハルシネーションによる事故や品質の揺らぎが発生する。

## 2. Decision (決定事項)
これらの問題を解決するため、以下の「物理的・論理的ハーネス（安全網）」をシステムアーキテクチャに導入する。

### 2.1 Dockerによる物理的サンドボックスの導入
`agent-core` のアクション（手足）がコマンドを実行する際、ホストOSのシェルを直接叩かせることを禁じ、専用の「Dockerコンテナ内」で実行させる。マウント領域を `workspaces/` のみに制限し、ホスト環境の破壊を物理的に不可能にする。

### 2.2 コアルールの「社長承認必須化」
`AGENT.md` などのコアルールはイミュータブル（一切変更不可）にはしない（運用に合わせて成長させるため）。ただし、これらのファイルを変更する Pull Request は、上位Agentによる自動Approveを禁止し、**必ず人間（社長）がレビューしマージする** ことを絶対のガバナンスルールとする。

### 2.3 クリティカル処理の `core-service` への委譲 (Service化)
API通信やDBアクセスなど「絶対に失敗してほしくない、品質が必要な処理」はAgentに都度作らせず、堅牢なDDDとTDDで守られた `core-service` に実装（Service化）する。AgentはそれらのServiceを「呼び出すだけ」に徹することで、処理品質の安定性を確保する。

### 2.4 依存性注入 (DI) によるシークレットの分離
`core-service` にはAPIキーなどのシークレットを持たせない。実際の環境変数やシークレットは `agent-core` 側で管理し、`core-service` のGateway呼び出し時に外部から注入（DI）する。

## 3. Consequence (結果)
- **メリット**: Agentの暴走によるシステム全体の破壊リスクが極小化される。また、クリティカルな処理がService化されることでAgentの作業品質が安定する。
- **デメリット・To-Do**: 隔離された開発作業用のDockerコンテナ環境を構築するタスク（Phase 2）が新たに追加される。
