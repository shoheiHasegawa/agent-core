## 💡 得られた教訓（Accumulated Wisdom）

### 1. Google Cloud / OAuth2 認証における新UIへの適応
* **新UIの構造**: Google Cloud の最新UI（Google Auth Platform）では、従来の「OAuth 同意画面」で管理していたテスト公開中の承認ユーザー管理が、左メニューの **「対象 (Audience)」** セクションに統合されていることを確認した。
* **認証時のセッション競合対策**: 開発環境から OAuth 認証（InstalledAppFlow）を起動した際、OSの標準ブラウザが自動的に別のアカウント（日常のアカウント）で認証を通そうとし、`エラー 403: access_denied` を引き起こすリスクがある。
  * **対策**: 発行された認証URLをコピーし、**「シークレットウィンドウ（プライベートブラウズ）」** に手動で貼り付け、明示的に開発用アカウント（`hs.app1ifeaut0@gmail.com`）でログインし直す手法が最も確実かつ堅牢である。

### 2. 人間心理に即したモーニングルーティンと時間設計
* **意志力（ウィルパワー）の保護**: 起床直後に「今日何をすべきか」を迷う時間が、二度寝や後回しを引き起こす最大の要因である。
  * **対策**: 朝の最初の15分（05:00 ~ 05:15）を「起動ジャーナリング（脳死で水を飲み、タスクを確認するだけの聖域）」としてブロックし、前日の未処理ジャーナルもここで吸収することで、行動へのハードルを極限まで下げる。
* **在宅/出社の補完的タイムライン構築**:
  * **月・金（在宅日）**: 通勤時間が不要なメリットを活かし、勉強（フォーカス時間）を終えた後（07:00 ~ 09:00）に「筋トレ＋準備（リフレッシュ）」を入れ、肉体を活性化させて9時始業へつなぐ。
  * **火〜木（出社日）**: 通勤を自動的なリフレッシュとし、会社に早く着くことで「08:00 ~ 09:00 の読書時間」を高集中状態で確保する。
  * **帰社日の金曜日**: 在宅日であっても出社を伴う日は自動的に「出社モデル」へ動的適用する例外ルールを YAML に記述。

## 🛠️ 技術的負債 (Tech Debt) と潜在リスク
1. **OAuthトークン（`token.json`）の寿命管理**:
   * 現在作成した接続検証スクリプトは `token.json` の読み込みと自動リフレッシュ（`creds.refresh`）を実装しているが、リフレッシュトークンが失効した場合、あるいはGoogleのポリシー変更によりセッションが切れた際、バックグラウンドバッチ（`launchd`）実行中にエラーが発生するリスクがある。
   * **対策**: バッチ実行時に認証エラーを検知した際、ユーザーの Obsidian 等に「再認可が必要」なタスクを自動注入し、アラートを上げるエラーハンドリングをアプリケーション層に設ける必要がある。
2. **秘密情報管理の境界**:
   * 今回は平文の `credentials.json` を使って検証を行った。リポジトリ側では SOPS で暗号化した `credentials.enc.json` をコミットしているが、ローカル実行時にはこれを復号化する、あるいは環境変数経由で `InstalledAppFlow` を構成するなどの「秘密情報とビジネスロジックの疎結合」を Phase 2 でさらに徹底する必要がある。

## 🚀 システム改善案 (Future Improvements)
1. **`Routines.md` の YAML バリデーション**:
   * COO Agent は将来的に `Routines.md` の YAML 構造体を Python からパースしてカレンダーを描画する。YAML 記述のシンタックスエラーや、キーの欠落を防ぐため、パース時に Pydantic などのスキーマ検証レイヤーをドメイン層（`domain/`）に定義し、不正な記述があってもシステムがクラッシュしないように防御する。
2. **週末の「タスクプール制」の自動家事トリガー**:
   * 毎週の必須家事（洗濯、掃除、買出）に加え、「風呂」「トイレ」「排水溝」といった4〜5週に1回の低頻度家事を、前回の実行記録（State）から逆算して、毎週土曜日の Drop_Zone に1つだけ自動注入する「ローテーションバッチエンジン」を application 層に設計・実装する。
