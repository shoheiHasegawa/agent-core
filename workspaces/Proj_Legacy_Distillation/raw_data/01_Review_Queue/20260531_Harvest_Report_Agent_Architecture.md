# Harvest Report: Agent駆動開発のアーキテクチャ刷新と教訓 (2026-05-31)

## 📌 概要 (Context)
COO (Task & Calendar Automation) モジュールの Phase 3 (YAMLパース実装) に入る前の設計レビューにおいて、AI（Agent）が局所最適に陥りやすい問題点と、TDDの厳格な適用方法について深いディスカッションが行われた。結果として、システム全体のドキュメント配置、フックの挙動、Agentの自律境界線に関するアーキテクチャ決定（ADR）が行われた。

## 🧠 教訓 (Wisdom & Insights)

### 1. ドキュメントの分散配置 (Colocation) と Agentic Routing
*   **問題**: すべての仕様や共通基盤のリストを中央（README等）に集約すると、Agentのコンテキストウィンドウが汚染され（Lost in the Middle）、推論精度が低下する。
*   **結論**: ドキュメントは【完全な分散配置（Colocation）】とし、各モジュールに閉じる。中央（`README.md` や `docs/INDEX.md`）には【次に何を読むべきかのポインター（薄い地図）】のみを置く。Agentは自律的に地図をたどり、必要な純度の高いコンテキストだけをフェッチする。

### 2. 人間の壁打ち (HITL) と 自律 (Autonomous) の境界線
*   **問題**: 仕様からいきなりAgentがテストコード（実装）を書き始めると、既存の共通基盤（Infrastructure）を無視した車輪の再発明（局所最適）が発生する。
*   **結論**: 【アーキテクチャの変更・エントロピーの増大】を伴う決定（例：既存の共通インフラを使うか、新設するか）までは人間が関与（設計レビュー）する。方針が固定化されたあとの【テスト起草〜実装〜リファクタリング】はAgentの自律に任せる。

### 3. TDDにおけるGitフックの最適解（例外のない穴の排除）
*   **問題**: TDDの「REDでコミットして仕様をロックする」思想と、「`pre-commit` でテストを強制する」思想が摩擦を起こす。WIPコミットの例外を設けるとAIのサボり（Sandbagging）の温床になる。
*   **結論**: `pre-commit` には Lint（静的解析）のみを残す。テスト（論理的品質）の強制はコミットフックから外し、AgentのSOP内のタスクと、最終防衛ラインである `pre-push` フックの2段構えとする。

## 🛠 技術的負債とアクションプラン (Tech Debt)
*   **Infrastructure層の不純化**: 現在、`infrastructure` の配下に `coo` や `finance` といったドメイン名のディレクトリが混在している。インフラ層は技術の詳細を隠蔽する場所であるため、今後は「技術名（`persistence`, `markdown`, `config` 等）」でディレクトリを切る形にリファクタリングを進める。
