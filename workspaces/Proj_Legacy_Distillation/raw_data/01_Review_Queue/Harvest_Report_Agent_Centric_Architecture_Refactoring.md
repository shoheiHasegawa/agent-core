# Harvest Report: To-Be ドキュメントアーキテクチャの純化と責務再定義

## 1. 概要 (Context)
Agent-Centricな「You, Inc.」アーキテクチャへの移行にあたり、To-Be（未来像）の設計書内に「過去の課題（As-Is）」や「移行手順」が混在し、エージェントのコンテキスト汚染を招いていた問題を解決したセッションの総括。

## 2. Wisdom (得られた教訓)
* **ドキュメントにおける「時間軸」の厳密な分離**
  * As-Is（過去/現在）、To-Be（未来）、Migration（橋渡し）、Decision（理由と経緯）の責務をディレクトリ単位で完全に分離しなければならない。To-Beドキュメントに「旧アーキテクチャでは〜」「〜を廃止し」といった比較表現を残すと、自律エージェントが「過去の仕様」を「現在の仕様」と誤認するハルシネーションの温床となる。To-Beは「あたかも既に完成しているかのように宣言的」に記述する必要がある。
* **物理的な制約（Local README）によるガバナンス**
  * システム全体の大枠ルールを定めるだけでなく、各ディレクトリ直下に `README.md` を配置し、「このディレクトリには何を書くべきか、何を書いてはいけないか（What & Bounds）」を明記することで、AIエディタ（Cursor等）や自律サブエージェントにコンテキストを強制認識させることができる。

## 3. Tech Debt (技術的負債・積み残し課題)
本セッションのレビューを通じて、移行および運用における以下の「技術的な穴」が明確化され、`open-item.md` へ積まれた。

* **内部リンクの破損とメタデータ変換**:
  * アーキテクチャの分割・ディレクトリ再編（フラット化）により、既存Markdown内のWikiリンクが大量に壊れる。移行時（Phase 3）に一括でリンクを修復し、新YAMLフロントマターを付与する自動化スクリプトの実装が急務。
* **iCloud同期とエージェント処理のコンフリクト**:
  * モバイルからの入力をAgentがバックグラウンドでパースする際、同期遅延や追記時のコンフリクトでファイルが破損するリスク。処理前に「安全なキューへMove/Lockする」トランザクション設計が必要。
* **NotebookLMの自動Syncの壁**:
  * Google DriveにKnowledgeをデプロイした後、NotebookLM側で自動同期をトリガーするAPIがない問題。手動Syncを運用フローに組み込むか、ローカルRAG代替案の検討が必要。

## 4. System Improvement (システム改善案)
* **WIP制限（Guardrails）のシステム的強制**:
  * `Sense_Making` に未処理タスクが滞留するのを防ぐため、Workspaceの作成をユーザーの手動フォルダ作成に任せず、HQ提供のコマンド（例: `make workspace`）経由で行わせ、CLI内で滞留数をバリデーションしてブロックする機構を設ける。
* **Quality Gateの自動化**:
  * 移行完了時や日次の運用において、Lintツールを用いて「リンク切れ（Broken Links）のゼロ確認」や「必須メタデータ漏れの検知」を機械的に保証する CI 的なパイプラインを構築する。
