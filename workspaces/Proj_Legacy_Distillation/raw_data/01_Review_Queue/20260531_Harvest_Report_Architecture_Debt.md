# Harvest Report: アーキテクチャ監査と負債の棚卸し (2026-05-31)

## 📌 概要 (Context)
COOモジュール Phase 4 のApplication層実装に際し、既存の `finance` や `knowledge` モジュールに蓄積している「技術的負債（DDD・クリーンアーキテクチャのアンチパターン）」についてアーキテクチャ監査を実施した。その結果と、今後のリファクタリングの指針となる「正しい設計原則」をここに記録する。

## 🚨 発見された技術的負債 (Tech Debt)

### 1. Application層の命名における「〇〇er」の横行
*   **事象**: `UrlConverterService` や `ReviewWatcherUseCase` など、機能的・機械的な命名が存在する。
*   **問題**: Application Service（ユースケース）は「システムがユーザーに対して提供する業務」の単位であり、「何を達成するか（動詞）」で命名されるべき。名詞（誰が・何が実行するか）に焦点が当たっていると、責務が肥大化しやすい。
*   **解決指針**: 今後は `ConvertUrlUseCase`, `WatchReviewUseCase`, `GenerateDailyScheduleUseCase` のように、動詞ベースの明確なトランザクション境界として命名する。

### 2. Port (Interface) の技術駆動命名と配置レイヤーの誤り
*   **事象**: `domain/finance/protocols.py` というファイルに、外部インフラへのインターフェース（`TransactionReader`等）が定義されている。
*   **問題**: 
    1.  `protocols.py` という名前はPythonの `typing.Protocol` に由来する技術駆動の命名である。
    2.  「外部からデータを読む・書く」という契約（Port）は、純粋なドメイン知識ではなく「ユースケースが外部に要求するもの」であるため、Domain層の汚染（DIP違反の温床）となる。
*   **解決指針**: 
    *   インターフェースは Domain 層ではなく **Application 層** に配置する。
    *   `ports.py` のようなアーキテクチャ用語のファイルすら作らない。**そのインターフェースを要求するユースケースのファイル内に直接 `Protocol` として定義する** のが最も凝集度が高くPythonicである（Pythonの構造的サブタイピングの利点を最大化する）。

## 💡 アクションプラン
*   今後の新規モジュール（COO等）の実装では、この「動詞ベースのユースケース命名」と「ユースケース内でのPort定義」を標準モデルケースとする。
*   既存の `finance`, `knowledge` 等のモジュールについては、今後改修が入るタイミングで順次この方針に従ってリファクタリングを実施する。
