# Harvest Report: COO Module Phase 1-4 総括 (2026-05-31)

## 📌 概要 (Context)
COO（Chief Operating Officer）モジュールの Phase 1（仕様定義）から Phase 4（Application層の実装）までが完了し、「ルーチン作業の読み取りと日次スケジュールの生成」という脳にあたる部分が完成した。
このプロセスで直面した設計上の課題と、今後のAIエージェント開発に活かすべき教訓（Wisdom）をここに総括する。

## 🏛️ アーキテクチャの進化と確立した原則

### 1. 純粋な技術とドメイン知識の完全分離 (Phase 3)
*   **課題**: 当初、Infrastructure層のリポジトリ（`routine_repository.py`）が特定のファイル（`Routines.md`）の形式やパーサー処理を密結合で抱え込んでいた。
*   **進化**: `MarkdownYamlParser` という純粋な技術モジュールを分離し、リポジトリ層はそれを「アダプター」として使う構成に改めた。これにより、インフラ層の中でも「純粋技術」と「ドメイン固有インフラ」が美しく分離された。

### 2. Application層の命名とFacadeパターンの採用 (Phase 4)
*   **課題**: 既存実装（`finance`等）に「〇〇er」という機械的な命名のユースケースが散見された。
*   **進化**: 
    1.  ユースケース名から機械的命名を排除し、`DailyScheduleService` のような「目的達成のためのFacade」として設計した。
    2.  Agentからの曖昧なリクエスト（「予定を立てて」「教えて」）を受け止める窓口として機能する。

### 3. Portの究極の配置 (Pythonic DIP)
*   **課題**: 外部インフラへのインターフェース（Port）を、`domain/protocols.py` のような技術駆動のファイルに置くアンチパターンが存在した。
*   **進化**: Pythonのダックタイピング（`typing.Protocol`）の特性を最大限に活かし、「Port専用のファイルすら作らず、そのインターフェースを要求するユースケースのファイル内に直接Protocolを定義する」方針を確立した。これが最も凝集度が高く、アーキテクチャ用語を排除できる究極の形である。
