# ⚠️ リスクアセスメントと導入要件 (Risk Assessment)

システムの導入にあたって想定されるリスク、制約事項、およびその対策を整理します。

---

## 📈 リスク判定まとめ（2026/04/20 更新）

| リスク項目 | レベル | 現状の分析と対策 |
| :--- | :---: | :--- |
| **後続プロセスとの断絶** | 🚨 高 | 既存のExcelを、他部署のマクロが直接参照している可能性。ヒアリング最優先事項。 |
| **マスタ管理の属人化** | ✅ 低 | **対策済み**: 「数式リセット機能（Idea-8）」の実装により、専門家不在でもワンクリック復旧が可能。 |
| **柔軟性の低下** | ✅ 低 | **対策済み**: 予定外日付アラートや自己修復機能の実装により、SS運用の脆さをカバー。 |
| **「隠れたルール」の喪失** | ✅ 低 | 管理者による例外操作は引き続き可能。システムMSG列での可視化も追加済み。 |
| **初期コスト・学習コスト** | ✅ 低 | Google環境に慣れた現場であれば移行コストは低い。 |

## 🔍 調査・確認すべき詳細項目

### 1. 外部連携（後続プロセス）の有無
*   **確認事項**: 誰かがこのExcelファイルを「ソースデータ」として別のExcelやBIツールに取り込んでいないか？
*   **影響**: もし自動連携されている場合、スプレッドシートへの移行でそのプロセスが停止する。

### 2. トラブル時の運用継続性（BPC）
*   **確認事項**: 作成者が不在の際、数式がズレたり不具合が出た際に運営が自力でリカバリできるか？
*   **解決案**: 「数式リセット機能」を搭載し、ジェネレータから再デプロイするフローをマニュアル化する。

### 3. イレギュラー対応の現行フロー
*   **確認事項**: 「当日キャンセル」「飛び入り」「遅刻の取り消し」など、Excel上でどう処理しているか？
*   **影響**: システム化によってこれらの操作が複雑にならないようなUI/ロジックの担保が必要。
