# トレード日誌自動化システム設計 (Mobile Thin-Wrapper Architecture)

## 1. 目的と背景 (Purpose & Background)

### 背景
*   **分断されたワークフロー**: これまでは楽天証券アプリでのスクショ取得、Geminiアプリでの壁打ち、Obsidianへの記録が分断されていた。
*   **「断面」の喪失**: 特に「チャート画像の喪失」により、後から振り返る際に最も重要な「その瞬間の判断の断面（インジケーターの状態や視覚的根拠）」が失われる課題があった。
*   **記録の障壁**: 手動でのコピペやファイル移動の手間が、継続的な記録（Second Brainへの蓄積）の障壁となっていた。

### 目的
*   **摩擦ゼロの記録**: トレード時の思考やリズムを一切途切れさせることなく、マルチタイムフレーム（週足・日足・30分足）の「判断の断面」をローカルのVaultへ確実に保存する。
*   **AI × 永続化の融合**: Geminiの高い対話能力を活かしつつ、データの永続化とバージョン管理（Git）を両立した拡張性の高いパイプラインを構築する。

## 2. システムアーキテクチャ概要 (What & How)

iPhoneのハードウェア（アクションボタン）を起点とし、iOSショートカットを「薄いUIのラッパー」として扱い、コアロジックをa-Shell上のPythonに委譲する構成。

*   **Trigger**: iPhone 16のアクションボタン（長押し）
*   **Frontend (UI / Routing)**: iOSショートカット（マスターメニュー経由）
*   **Backend (Logic / File OS)**: a-Shell + Python 3 (`generate_trade_log.py`)
*   **Analysis (AI)**: Gemini 公式アプリ
*   **Storage & Sync**: iCloud (SOURCE_DIR) → 拡張版 `mobile-sync` (launchd) → ローカルPC（Obsidian Vault）

## 3. 主要な決定事項とその根拠 (Decisions & Why)

### 決定1: アクションボタンとマスターメニューによる起動
*   **内容**: 楽天証券アプリを開いたまま、アクションボタン長押しでマスターメニューを展開し、そこから処理をキックする。
*   **根拠**:
    *   **摩擦の排除**: アプリの切り替えや写真（ギャラリー）からの手動選択をなくし、トレード時の思考の分断を防ぐため。
    *   **拡張性の担保 (Facadeパターン)**: アクションボタンをメニュー化することで、将来的な機能追加（例：ヘルスケアデータの記録など）の入り口として機能させる。

### 決定2: Thin Wrapper（Python委譲）アーキテクチャの採用
*   **内容**: iOSショートカットには「最新のスクショ3枚の取得・一時保存・Geminiへの共有」のみを行わせ、ファイルのリネームやMarkdownテンプレートの動的生成はすべてa-Shell上のPythonスクリプトで行う。
*   **根拠**:
    *   **Git管理の実現 (Docs as Code)**: 複雑なロジックをPythonに切り出すことで、PC（AntiGravity）で開発・Git管理し、iPhoneで `pull` するだけでデプロイできる環境を維持する。
    *   **保守性の向上**: iPhoneの小さな画面でショートカットの複雑な条件分岐を組む苦痛を排除する。

### 決定3: Gemini公式アプリへの「共有（Share）」による直接連携
*   **内容**: APIを使わず、ショートカットから取得したスクショ画像を直接Gemini公式アプリに共有して起動させる。
*   **根拠**:
    *   **コストと性能**: APIコストをゼロに抑えつつ、Proモデルの恩恵を最大限に受ける。
    *   **役割の分離**: 「データの確実な保存（iCloud）」と「リアルタイムの思考の壁打ち（Gemini）」を分離し、GeminiのUI制約を回避する。

### 決定4: 既存システム `mobile-sync` の画像転送への拡張
*   **内容**: iCloudに生成された `.md` ファイルだけでなく、`attachments` フォルダ内の画像（`.png`）もローカルPCへ一方向同期（移動）させるよう、既存のBashスクリプトを改修する。
*   **根拠**:
    *   **資産の有効活用**: すでに安定稼働している自動搬送路（launchd）を活かし、スマホ側のストレージを圧迫しない運用を継続する。

## 4. 実装詳細と運用 (Implementation & Operations)

### 4.1 パス構成
*   **iPhone (a-Shell / Obsidian)**:
    *   `Mobile_Vault/` (Vaultルート)
    *   `Mobile_Vault/01_Inbox`: スクショの入力元 兼 日誌の生成先
    *   `Mobile_Vault/99_System/Attachments`: 画像の移動先
    *   `Mobile_Vault/99_System/Scripts`: Pythonロジック
    *   `Mobile_Vault/99_System/Templates`: 日誌テンプレート
*   **PC (Local / Git)**:
    *   `second-brain/01_Inbox`: Markdown同期先
    *   `second-brain/99_System/Attachments`: 画像同期先
### 4.2 運用フロー (トレード時)
1.  **撮影・アノテーション**: 週足 → 日足 → 30分足の順にスクショ。その場でiOSの描画機能でチャートに気づきを書き込む。
2.  **手動保存**: iOSの「"ファイル"に保存」メニューから、`Mobile_Vault/01_Inbox` を指定して保存。
3.  **自動化トリガー**: アクションボタンからショートカットを起動。
    *   Pythonが画像をリネーム・移動し、日誌を生成。
    *   ショートカットが最新の3枚（アノテーション済み）を取得し、**クリップボードにコピー**。
    *   **Geminiアプリを自動起動**。
4.  **議論**: Geminiの入力欄にペーストし、分析の壁打ちを行う（※将来的にGemを活用）。

## 5. 設計の変遷と意思決定の履歴 (Development History)

### 2026-04-27: 構造の最適化と運用スタイルの確立
*   **Mobile_Vaultの独立**: a-ShellとObsidianの親フォルダを一本化するため、専用のVaultディレクトリをiCloud上に構築。
*   **Scripts/Templatesの分離**: ロジック（Python）と意匠（Markdown）を分離。Obsidian側からテンプレートを直接編集可能な構成に変更。
*   **アノテーション運用の採用**: 「自動保存」よりも「思考の言語化（書き込み）」を優先。スクショ撮影直後にユーザーが `01_Inbox` へ手動保存するフローを正装とした。
*   **絶対パスへの移行**: a-Shell上の接続名に依存しないよう、スクリプト自身がVaultルートを自動検知するロジックを実装。
*   **クリップボード連携ハック**: iOSの共有シートの制約（複数画像共有の非対応）を回避するため、「クリップボードコピー ＋ アプリ起動」のフローを採用。結果的に共有シートの選択工程が省かれ、UXが向上した。

### 2026-04-29: ソート順序の堅牢化と仕様の明文化 (SDD移行)
*   **数値プレフィックスの導入**: iOSショートカット側で「名前：ZからA」でソートする際、ラベル名のアルファベット順（weekly/daily等）に依存せず確実に「週足 -> 日足 -> 30分足」の順でクリップボードに並ぶよう、ファイル名に `3_weekly`, `2_daily`, `1_30m` という数値プレフィックスを付与。
*   **仕様駆動開発 (SDD) への移行**: プログラムの技術的詳細（デプロイフロー、a-Shell制約等）を `daily-tools` 側の [spec.md](file:///Users/shoheihasegawa/play_ground/daily-tools/trading-automation/spec.md) へ分離。本ドキュメントは人間向けの「設計思想・全体像」を管理する役割に特化。
*   **UXフローの再定義**: アクションボタン長押しから、Geminiへのペースト、そして `mobile-sync` によるPCへの自動搬送までの一貫した「摩擦ゼロ」の体験を、数値プレフィックスという小さなハックで支える構造を確立。
