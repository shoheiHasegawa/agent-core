# AI駆動型プロジェクト管理・ナレッジ運用アーキテクチャ（Permanent Note 原案）

AIエージェント（AIオーケストレーター）をチームの一員として動かすための、次世代のプロジェクト管理とナレッジ運用のアーキテクチャ構想。

## コアコンセプト：人間は「オーケストレーション」に集中する
- スプレッドシートや議事録の体裁調整など「人間がやるべきでない雑用」を徹底的に排除。
- 人間はAIに対する「目標・制約条件・評価基準」の設計に集中する。
- 情報の **Single Source of Truth** をGitリポジトリ（主にMarkdown）に集約し、仕様書・コード・タスクを密結合させる。

## 具体的なアーキテクチャとワークフロー

### 1. インプットの効率化（エディタ直結のメモ）
- Googleドキュメント等は使用せず、Gitをチェックアウトした状態で Cursor や VSCode などのエディタから直接 `.md` にメモを取る。
- 会議中は体裁を気にせず書き散らす。

### 2. AIによる情報の構造化と課題抽出
- 乱雑なメモに「テンプレートプロンプト」を適用し、AIエージェントに「決定事項」「仕様」「直近の課題」を自動抽出・構造化させる。
- 抽出済みの元メモはアーカイブ（ゴミ箱やDrive）に退避。

### 3. 完全自動のタスク管理
- スプレッドシート管理を廃止。GitのIssueをマスタとする。
- AIに抽出させた課題から、API等を経由してIssueを自動起票。課題一覧のカンバンやテーブル（Markdown）もAIが自動更新する。

### 4. 仕様駆動開発 (OpenSpecの概念)
- ソフトウェアの振る舞いや前提を `.md` に定義（OpenSpec）し、Gitで管理する。
- Devin等のAIエージェントに対して特定ブランチを指定し、**「コード修正」と「ドキュメント更新」をセットで実行** させることで、仕様書の陳腐化を完全に防ぐ。

### 5. ステークホルダーへの展開とQA
- 業務部門へはGit上のMarkdownを自動変換してGoogleドキュメント等へ共有する。
- すべての仕様や議事録（Markdown群）を **NotebookLM** などのRAGに読み込ませ、全体文脈を保ったQA対応ボットとして機能させる。
