# Operations Workflow Policy

## オペレーション・フロー（日々のルーティン）

### Phase 1: 収集と発生（Daily Workflow）
1. **`01_Inbox/fleeting` への記述**: 思いつき、タスク、作業ログは、迷わず fleeting ノート（または一時ノート）に時系列で書く（散らかって良い）。
2. **切り出し（Refactor）**: 独立したトピックだと感じた塊は、その日か翌朝に別ノートで切り出し `01_Inbox` 直下（または適切なフォルダ）へ送る。

### Phase 2: 整理と配置（Organize Workflow）
定期的に `01_Inbox` をゼロにする。
1. 特定の期限・ゴール付きの仕事は `10_Projects` へ。
2. 日常の維持管理・R&Dマニュアルは `20_Areas` の各部門へ。
3. **外部ソース（Kindleハイライト等）の特例**: 同期事故を防ぐためInboxへは入れず、最初から `30_Resources/Literature` などに直接「図書館式」で保存し、タグで未読フラグを管理する。

### Phase 3: 知識の醸成（Thinking Workflow）
1. `01_Inbox` に残った知識や読書メモは「自分の言葉」で書き直し、文脈を切り離して `30_Resources/Permanent` に格納する。
2. **AIアシスタントの活用**: `/zettelkasten` ワークフロー等を用いて「壁打ち（Socratic Dialogue）」を行い、受動的な知識から能動的な知識創造へ転換する。

### Phase 4: 完了時の収穫（Harvesting）とアーカイブ
1. プロジェクト（`10_Projects` 内）の完了時、`40_Archives` に移す前に事務連絡を捨て、**「教訓」「SOP」「テンプレート」** を抽出する。
2. 抽出した知識資産を `30_Resources` または各C-Levelディレクトリに還元させる。

**【Archivesの格納ルール】**
- **PARA階層の厳守**: `40_Archives` 直下には `10_Projects`, `20_Areas`, `30_Resources`, `99_System` のみを置き、独自のカテゴリ（AIなど）を作らない。
- **階層のフラット化**: `10_Projects` 内のプロジェクトをアーカイブする際は、元のカテゴリ階層（Development等）を維持せず直下にフラットに配置する。
- **時系列プレフィックス**: 検索・ソート性を高めるため、アーカイブ時のファイル/フォルダ名には必ず `YYYY-MM-DD_` を付与する。
- **ゴミの排除**: AI等による自動生成ログ（処理済みのReview等）は、知識抽出が完了していればアーカイブせずに破棄（削除）する。

### Phase 5: 市場へのデプロイ（Export）
- `30_Resources` に組み上がった永久保存版メモを素材とし、SNS等のコンテンツ（アウトプット）へ変換し、知識の回転率・市場価値を高める。
