# 07. Reverse Recovery Flow (逆方向の実績回収) 要求・要件定義書

本ドキュメントは、実績回収から分析・改善に至るまでの「データフローと蓄積」に関する要求（Why/What）と、それに基づくシステム要件・アーキテクチャ（How）を定義します。

---

## 1. 要求事項 (Requirements)

### 1.1 改善フロー（PDCA）を回すための実績蓄積
- **要求**: 月次・週次のトレンド分析や `10_Areas` ごとのリソース配分改善を行うため、タスクの完了実績や投下時間を永続化したい。
- **解決の方向性**: 「アクティブ（これからやるタスク）」と「歴史（完了済みのログ）」でデータの性質を分け、歴史データは集計に特化した「データウェアハウス（SQLite）」へ移す。

### 1.2 アーキテクチャと責務の純化
- **要求**: システムデータと、人間の知識（Zettelkasten）の境界を明確にする。
- **解決の方向性**: 生のログデータは `agent-core` 側に閉じ、意味づけされたインサイト（学び）のみを `second-brain` 側に配置する。

---

## 2. システム要件・アーキテクチャ定義 (Architecture Decisions)

### 2.1 蓄積場所（Data Warehouse）の技術選定
- **データストア**: **SQLite (ローカルDB)**
  - 理由: `10_Areas` ごとの集計や月次推移の分析において、SQLによる集計クエリが必須なため。
- **配置先**: `agent-core/data/metrics.db`

### 2.2 記録すべき情報（DBスキーマ設計と制約）
DBへのINSERT時、以下の情報を記録します。バッチ重複実行時の二重計上を防ぐため、**UPSERT（冪等性確保）** を必須とします。
- `task_id` (Task/Routine固有のID)
- `title`
- `task_type` (単発/Routine/Recurring)
- `category` (M/S/W)
- **`area_id`** (10_Areasのどこに属するか)
- `estimated_minutes` (見積もり時間・分単位)
- `actual_minutes` (実績時間・分単位)
- **`worked_date`** (稼働日/論理日)
- `is_completed` (BOOLEAN: その日の作業でタスク全体が完了したか)
- **制約要件**: データの重複を防ぐため、`task_id` と `worked_date(論理日)` の複合UNIQUE制約を設ける。

### 2.3 タスクのライフサイクル（Soft Deleteとタスク種別分岐）
タスク種別によって完了時のライフサイクルを分岐させます。誤操作対策として物理削除は行いません。

1. **One-off Task (単発タスク)**:
   - DB上の実績テーブルに記録後、`you_inc_ops.db` のタスクレコードのステータスを完了に更新する（Soft Deleteによる論理アーカイブ。物理ファイル移動は行わない）。
2. **Routine / Recurring Task (繰り返しタスク)**:
   - DB上の実績テーブルにその日の実績（1回分のログ）をINSERTし、親となるタスクレコード自体は維持する。
   - 代わりにレコード内の `last_completed_at` や `current_streak` などのステータスを更新する。

### 2.4 Weekly Review の出力形式と配置
- **配置先**: `second-brain/20_Sense_Making/`
- **命名規則**: `Weekly_Review_2026-W28.md` （フラット配置）
- **テンプレート**: 出力時は `second-brain/90_Meta/Templates/Sense_Making_Template.md` に準拠し、Zettelkastenのルールを厳守する。

## 3. 運用・エッジケース要件 (Edge Cases & Operations)

### 3.1 実績時間の回収UIと抽出ルール（単位と状態の分離）
ダッシュボード（`Briefing.md`）上のUIはシンプルに保ち、複数日タスクの累計時間の管理などは裏側のSQLite DB（`you_inc_ops.db`）内でのみ保持します。

- **出力ルール**: 朝のバッチ (`generate_daily_briefing.py`) は以下のように、その日のタスクとメモ領域を出力する。
  `- [ ] 〇〇の資料作成 <!-- id: task_123 -->`
  `  前回メモ: (前日のメモがあればここに表示)`
- **ユーザーの入力ルール (Super-loose Parser)**:
  1. **作業したが終わらなかった場合**: `[ ]` のままで、行末に稼働した分数を書く（例: `45`）。
     `- [ ] 〇〇の資料作成 45`
  2. **作業して完了した場合**: `[x]` を付ける。実績時間を書く場合は行末に追記する。
     `- [x] 〇〇の資料作成 90`
  3. **メモを残す場合**: タスクの次の行に `メモ: 〇〇` と記載する。
     `メモ: Aまで終わった。明日続きから。`
- **抽出・パース・フォールバックルール (`sync_worklogs.py`)**:
  - `[x]` または行末の数値を解析し、その日の `actual_minutes` としてDBへUPSERTする。
  - **安全装置（フォールバック）**: `[x]` 完了マークがついているにもかかわらず、実績の書き換えを忘れて行末が空だった場合、パーサーは `estimated_minutes`（予定）の数値を自動的に実績として計上する。
  - `メモ: ` を検知した場合、DBの `last_memo` カラムを更新し、翌朝の自動出力時に `前回メモ: ` として引き継ぐ。
  - `[x]` が付いている場合のみ、Task Registry側でタスク全体を `status = COMPLETED` に更新し、アーカイブ処理（Soft Delete）へ回す。

### 3.2 論理日の絶対指定とバッチ実行遅延対策
- **要件**: 実行時刻のズレによる計上ミスを防ぐため、論理日（`worked_date`）の判定はシステム時刻(`time.Now()`)に依存させず、**パース対象の `Briefing.md` ファイル名（例: Briefing_2026-07-18.md）が持つ日付文字列を絶対的な正（Target Date）**としてDBへ記録する。

### 3.3 手動修正と誤操作（Undo）対応
- **時間修正**: DBのレコードを手動編集するため、LLMエージェントに指示すればAgentがSQLを叩いて修正（データパッチ）できる仕組み（Tool）を提供する。
- **誤完了の運用**: 誤って `[x]` を付けてバッチが回ってしまった場合、システム的な「Undo（復元）機能」は実装**しない**。そこまでの投下時間はDBに正しく計上されておりデータ損失がないため、ユーザーは単純に**「〇〇（残り）」という新規タスクを作り直す運用**とし、システムをシンプル（ステートレス）に保つ。
