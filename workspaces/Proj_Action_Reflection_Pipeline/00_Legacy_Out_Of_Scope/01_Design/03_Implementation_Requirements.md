# 実装要件定義書 (Implementation Requirements)

本ドキュメントは、別セッションのAgentが迷いなく「PM Agent / Coach Agent」の基盤実装（PoC）に入れるように、システム要件とインターフェース仕様を定義するものである。

## 1. 開発環境と技術スタック（推奨）
*   **言語/ランタイム**: Python (Google Calendar APIとの親和性、LLM SDKの扱いやすさを考慮) または Node.js。次セッションのAgentが最適なものを選択可。
*   **実行基盤**: Macの `launchd` (plistファイルで管理)
*   **LLM API**: OpenAI API (GPT-4o) または Gemini API (複雑な自然言語の構造化用途)。
*   **カレンダー連携**: Google Calendar API (Service Account または OAuth2.0 Desktop app 認証)。

## 2. ディレクトリ構造とパス
*   **Input Dir 1 (動的メモ)**: `~/play_ground/second-brain/00_Inbox/Drop_Zone/`
    *   ここに `.txt` または `.md` ファイルがiOSショートカット経由（iCloud Drive同期など）で格納される。
    *   *【重要】Mac起動直後はiCloudの同期ラグが発生する可能性があるため、PM Agentの実行はMac起動から数分後、またはジャーナリング完了後に行う設計とすること。*
*   **Input Dir 2 (静的ルーティン)**: `~/play_ground/second-brain/10_Projects/Task_and_Calendar_Automation/Routines.md` (または同等の特定ファイル)
    *   ユーザーが定義した毎月・毎週のルーティンタスクのリスト（自然言語記述）。
*   **Output Dir (Obsidian)**: `~/play_ground/second-brain/` 以下の特定フォルダ。

## 3. 実装対象のコンポーネント (MVPスコープ)

### 3-1. Phase 0: 既存カレンダーからのデータ移行スクリプト（Migration）
新仕様ではSecond Brainが全ての正本となるため、既存のGoogleカレンダーに残っている「未来の予定」がノイズになります。これを解消するため、実装フェーズの一番最初に以下のワンオフ（使い捨て）スクリプトを作成・実行します。
1.  **抽出**: Google Calendar APIを用いて、今日以降の未来の予定（繰り返し設定含む）を全て取得する。
2.  **変換**: LLMを用いて、取得した予定を「ルーティン（定例会議など）」と「単発の未来予定（歯医者など）」に分類し、Obsidian内の `Routines.md` 等にMarkdownのテキストとして書き出す。
3.  **パージ**: 抽出が完了した未来の予定をGoogleカレンダーから一括削除（または別カレンダーへ退避）し、Agentが描画するための真っ白なキャンバス（Hub）を用意する。

### 3-2. iOS Shortcuts (ユーザー構築支援)
*   PC側のAgentからは直接iPhoneを操作できないため、実装フェーズの最初に**「ユーザーがiPhone上で設定するためのショートカット構築手順書」**を生成して提示すること。
*   **必要なショートカット**:
    1.  `Quick Memo`: 音声・テキストを受け取り、`Drop_Zone` に `yyyymmdd_HHMM_memo.txt` として保存。
    2.  `New Task`: 明日以降のタスクを素早く入力するための専用口。

### 3-2. PM Agent Core (バッチスクリプト / Skill)
1.  **実行トリガーの2段構え**: 
    *   【能動】Antigravity上でのCoach Agentジャーナリング完了時にキックされるSkillとしての実行。
    *   【受動】ジャーナリングがスキップされた場合のセーフティネットとしての `launchd` による自動実行（23:00等）。
2.  **データの読み込み**: 
    *   `Drop_Zone` 内のテキストファイルを全て読み込む。
    *   ルーティンのマスターファイル（`Routines.md`）を読み込む。
    *   カレンダーの「固定予定（繰り返し予定）」と「実績」を読み込む。
2.  **LLM Parsing**: テキストの内容から「タスク」「日時」「優先度」「コンテキスト」を抽出するプロンプト処理。
3.  **自律的成長と休息のスケジューリング**:
    *   Agent自身のプロファイルファイル（`PM_Agent_Wisdom.md` 等）から、ユーザーの過去の傾向（タスクのこなせる粒度や疲労度）を読み込む。
    *   タスクを適切な粒度に分解し、必要に応じて**「空白の時間（休息）」をGoogleカレンダーに強制的にブロックする**。
4.  **Google Calendar API**: 
    *   「明日の予定」を取得（空き時間の確認）。
    *   抽出・分解したタスクを「明日の空き枠」に登録し、詳細（Description）に `obsidian://open?vault=second-brain&file=...` のディープリンクを埋め込む。
5.  **Markdown出力と自己更新**: 
    *   タスクの詳細コンテキストをObsidian内のMarkdownへ出力する。
    *   処理後、得られた教訓（タスクが多すぎた等）があれば、Globalルールに基づき自身のプロファイルファイルの `Accumulated Wisdom` を自己更新する。

### 3-3. Coach Agent (プロンプト・ジャーナリング基盤)
*   単なるタスクの振り返りではなく、「未来に向かって迷子になっていないか」「作業が目的化していないか（メンタルブロック）」を問いかけるシステムプロンプトの設計。
*   **UI要件（Antigravity Skill）**: 
    *   AntigravityのチャットUI上で動く「Skill」としてジャーナリングを実装する。
    *   ユーザーにはカレンダーで「1日の振り返り」予定（リマインダー）を設定してもらい、時間になったらAntigravityを開いてSkillを起動し、対話を行うルーティンとする。
    *   対話内容は後から振り返れるように、最終的にObsidianのデイリーノート等にMarkdownとして追記・保存される仕組みを設ける。

## 4. 次のAgentへの引き継ぎ事項
*   本プロジェクトのゴールは「単なる自動タスク登録」ではなく、「ユーザーの認知（世界観）を構築し、摩擦ゼロで実行にフォーカスさせる」ことにある。
*   UIの実装（モバイルアプリ開発）は行わず、**データのルーティングと変換（LLM）** に徹すること。
*   まずは `Drop_Zone` のテキストを読み込んで、Google Calendar に反映させる一連のシンプルなPythonスクリプト（PoC）の作成から着手すること。
