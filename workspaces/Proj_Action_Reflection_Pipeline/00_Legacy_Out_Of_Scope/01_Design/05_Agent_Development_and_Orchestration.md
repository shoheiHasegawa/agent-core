# Agent Development & Orchestration Architecture

本ドキュメントは、「Task and Calendar Automation (Agent OS)」プロジェクトにおける、**AIエージェント自身を用いた自律開発の手法と、アーキテクチャの運用ルール**を定義するものです。

人間のシステム開発者が守るべきルールだけでなく、「Agentがコードを書く際に遵守すべきセルフ・オーケストレーションの作法」を明文化しています。

## 1. 知識 (Knowledge) と 手順 (Procedures) の完全分離

エージェントが迷いなくタスクを実行し、かつシステムが陳腐化しないための大原則として、**「ルール（何を作るか・なぜ作るか）」と「プロセス（どう作るか）」を物理的に別々のディレクトリに分離**して管理します。

*   **`docs/` (またはPJディレクトリ)**: 【知識の正本管理】
    *   システムアーキテクチャ、コーディング規約、失敗事例（アンチパターン）などの「不変・中長期的なルール」を配置する。
    *   Agentのプロンプト（`.agent/` 内）に直接知識を書き込んではならない。
*   **`.agent/workflows/` & `skills/`**: 【手順の定義】
    *   複数のタスクを束ねるオーケストレーション（Workflow）と、単一タスクの実行（Skill）を定義する。
    *   これらの中には「Step 1: `docs/architecture.md` を読み込め」という**参照の指示だけ**を持たせる。
*   **`AGENTS.md`**: 【地図】
    *   ディレクトリのルートに配置され、Agentが最初に読み込むべき「Index（地図）」の役割を果たす。知識と手順の置き場所をマッピングする。

## 2. Agent Orchestration (Sub-Agent Review Board)

LLM（エージェント）単体で大規模な実装とレビューを兼任させると、必ず「ハルシネーション」や「局所最適化（テストを通すためだけのダーティハック）」を引き起こし、アーキテクチャが急速に劣化します。
これを防ぐため、本プロジェクトでは**「実行」と「レビュー」を別々のサブエージェントに強制分割（`invoke_subagent`）させる Review Board パターン**を採用します。

### 🚨 陥りやすいアンチパターン (Single-Thread Trap)
*   一人のAgentに「ドメインロジックの追加とテスト」を丸投げすると、インフラ層（Repositoryなど）に複雑なIF文やマッピング処理を直接書き込んでしまう（SRP違反・ドメイン知識の漏洩）。
*   自身が書いたコードの欠陥は、自身では気づけない。

### 🛡️ 解決策 (TO-BE Workflow)
開発ワークフロー（例：`.agent/workflows/software_development.md`）において、実装ステップの完了後、**必ず独立した「Reviewer Sub-agent」を生成し、アーキテクチャ監査（SOLID原則・DDD準拠の確認）を行わせること**。
Reviewerからの指摘があった場合は、人間ではなく「Implementer Sub-agent」を再度生成して修正にあたらせる。

## 3. SOLID / DDD における Agent のコーディング責務

Agentが `life-automation`（Python等のシステム側）のコードを書く際、以下のドメイン駆動設計（DDD）の原則を絶対遵守すること。

*   **「特定の知識」と「汎用メカニズム」の分離**:
    *   「第3金曜日は帰社日である」「Category A は赤色で表示する」といった**ユーザー固有の生活ルール（特定の知識）は、すべて `second-brain` 側（MarkdownやYAML）で保持する**。
    *   システム（`life-automation`）のインフラ層にこれらをハードコードしてはならない。
*   **Domain Service / Factory によるカプセル化**:
    *   外部から渡されたルール（知識）を評価・計算する処理は、必ず `domain/` 内の Service や Factory クラスに抽出する。
    *   インフラ層（Repository等）は、純粋にデータを「取得・保存」するだけの Dumb（愚か）な層として維持し、ドメインの解析ロジックを一切混入させないこと。
